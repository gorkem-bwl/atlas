## The test situation

### Existing test suite is partially broken (PRE-EXISTING, not v1.10-caused)

**Verified by running the exact same commands against v1.9.5**: failure
counts are identical at both commits. v1.10 introduced zero regressions.

Current state:
- `cd packages/server && npm test` → **11 files fail, 55 tests fail,
  174 tests pass (76%)**
- `cd packages/server && npm run test:integration` → **3 files fail,
  12 tests fail, 21 pass (64%)**
- `cd packages/client && npm test` → **2 files fail, 15 tests fail,
  406 pass (96%)**
- `e2e/` playwright specs exist but not run in this audit

### Root cause of the pre-existing failures

1. **Server unit tests**: CRM, HR, Docs, Drive, Tasks, Draw, Projects test
   files all use `vi.mock('../src/apps/X/service')` to mock a barrel file,
   but the service layer was **split into entity-domain sub-services in
   commit `cb9abf9` refactor: split server controllers and services into
   entity-domain modules**. Controllers now import directly from the
   sub-services (e.g. `../services/deal.service`), so the mocks never
   intercept. Every call reaches the real service which then fails tenant
   lookups against mock data. Fix per file: re-declare the `vi.mock()`
   against each sub-service path.

2. **`packages/client/test/hooks-projects.test.ts`**: tests expect
   `useInvoices`, `useInvoice`, `useCreateInvoice`, `useUpdateInvoice`, etc.
   to be exported from `apps/projects/hooks.ts`, but those hooks were
   **moved to `apps/invoices/hooks.ts` in commit `29141cc` feat: rewire
   Projects to use CRM companies and shared Invoices**. Fix: remove the
   invoice-related tests from `hooks-projects.test.ts` and either delete
   them or port to a new `hooks-invoices.test.ts`.

3. **`packages/client/test/ui-sidebar.test.tsx`**: 2 tests assert specific
   color values for active sidebar items that drifted from current theme.
   Fix: update the expected hex codes OR assert against CSS variables.

4. **Server integration tests (CRM/Drive/Tasks)**: 12 failures. Haven't
   root-caused yet — likely the same barrel-split problem or DB fixture
   rot. `auth`, `settings`, `system` integration tests pass.

**Phase B — add v1.10 unit tests**
Write focused unit tests (~30-40 tests, ~4-5 hours of work) for:

1. **`payment.service.ts`** — most important:
   - `recordPayment` validation (amount > 0, not in future, not over total,
     refund not exceeding paid)
   - `recordPayment` status transitions (sent → paid on threshold)
   - `updateInvoicePaidStatus` revert (paid → sent on refund, paid →
     overdue on refund past due date)
   - `updatePayment` immutable fields (type, invoiceId silently ignored)
   - `deletePayment` recomputation
   - Transaction + row lock semantics (mock db.transaction)

2. **`recurring-invoice.service.ts`**:
   - `addFrequency` helper — weekly/monthly/quarterly/yearly correctness,
     month boundary edge cases (Jan 31 → Feb 28)
   - `generateInvoiceFromRecurring` happy path + deactivation on maxRuns
     hit + deactivation on endDate past
   - Update path recomputes nextRunAt correctly

3. **`reminder-scheduler.ts`**:
   - `computeEligibleStage` pure function — stage 0→1, 1→2, 2→3, 3→4+
     (endless), never-fire-when-paid
   - Edge: stage 3 with null lastReminderAt defensive fallback

4. **`invoice-email.service.ts`**:
   - Template branching (invoice vs reminder vs payment)
   - Recipient resolution chain (override → primary contact → invoice contact)
   - Portal token hard-fail when null
   - Never-throw contract (all failure paths return `{sent: false, reason}`)

5. **`email-templates.ts`**:
   - All 3 template builders produce non-empty text + html + subject
   - customSubject overrides default
   - customMessage injected
   - Stage-specific subject strings match spec
   - companyTaxId rendered in footer
   - accentColor default fallback

Use the existing vitest setup and `vi.mock` pattern. Avoid hitting a real
DB — mock `db` and work with return values.
