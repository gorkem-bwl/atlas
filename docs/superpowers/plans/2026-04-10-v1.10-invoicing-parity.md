## Goal

Close the top 5 functional gaps that currently prevent Atlas Invoices from
being usable as a small business's primary billing tool. After this ship,
a solo consultant or small agency should be able to:

1. Track time against a project
2. Pull those hours into an invoice with one click
3. Email the invoice (with PDF attached) to the client from inside Atlas
4. Record when the client pays (full or partial) and see accurate balance
5. Set up a monthly retainer/subscription that generates invoices
   automatically
6. Have Atlas nudge overdue customers on their behalf via scheduled
   reminder emails

Source: [competitive-analysis-invoiceninja.md](../../competitive-analysis-invoiceninja.md)
(gap analysis that identified these 5 as the highest-value missing
features).

## Non-goals (deferred to later releases)

- **No payment gateway** (Stripe/PayPal/etc). Weeks of work each. v1.11+
- **No auto-bill** (charging stored cards). Depends on gateway.
- **No credit notes**. Tripled complexity for a use case Atlas users
  rarely hit. v1.11+.
- **No subscription / one-click buy pages**. Needs gateway + portal flow.
- **No multi-currency FX conversion**. Currency stays a display label.
- **No compound tax / tax inclusive vs exclusive**. Per-line tax as-is.
- **No late fee auto-application**. Needs payments table (yes, included) +
  accounting decisions (no). v1.11+.
- **No quotes inside Invoices app**. CRM Proposals stays a separate flow.
- **No custom fields on invoices**. Separate initiative.
- **No pay-invoice action in client portal**. Portal stays view-only.
- **No file attachments on invoices** (beyond the auto-generated PDF).
- **No API / webhook surface for third parties.**
- **No per-client defaults or client groups.**

If any user request during execution suggests building one of these,
defer and add it to `docs/superpowers/plans/v1.11-invoicing-followup.md`.

## Decisions already made (do not re-litigate)

These were confirmed with the user during planning. Execute without
asking again.

1. **Payments model is standard double-entry**: one invoice can have many
   payment records; invoice `balanceDue = total - SUM(payments.amount)`.
   Not a single boolean flag.

2. **Refunds are included** in v1.10 (but credits are not). A refund is a
   payment with a negative amount OR a separate row with a `type: 'refund'`
   discriminator. The plan uses a `type` enum ('payment' | 'refund') on the
   payments table — clearer than signed amounts, easier to report on.

3. **SMTP infrastructure already exists.** `sendEmail()` in
   `packages/server/src/services/email.service.ts` is wired to SMTP
   settings stored in `system_settings` via the System > Email admin UI.
   HR Expenses already uses it. **No new email infrastructure is needed** —
   just new templates + invoice-specific send paths. If the user hasn't
   configured SMTP in their System panel, `sendEmail()` returns `false`
   gracefully with a debug log, and the calling code handles that.

4. **Reply-to address is tenant-level**, pulled from `invoiceSettings`
   (which already has `companyEmail`). If that's empty, fall back to the
   SMTP `from` address (already configured in System > Email). Never use
   individual user emails — multi-tenant leakage risk.

5. **Scheduler pattern is the existing `setInterval`** approach used by
   Sign reminders, Task reminders, CRM activity reminders, and HR leave
   balance. Do NOT bring in BullMQ or any new queue runtime. New schedulers
   export `startXxxScheduler() / stopXxxScheduler()` and get wired from
   `packages/server/src/index.ts`. Model the invoice reminder scheduler
   on `packages/server/src/apps/sign/reminder.ts` (closest cousin).

6. **Recurring invoices support both modes**: per-recurring-template a
   boolean `autoSend` flag. Default: **false** (creates a draft invoice
   on the schedule, user reviews and sends manually). If `autoSend: true`,
   the generated invoice is immediately emailed to the client using the
   send flow from phase 1.

7. **The cron granularity is hourly** for reminders and daily for
   recurring invoice generation. Recurring invoices don't need
   sub-hourly precision; reminders are cheap enough to check hourly.

8. **Invoice "sent" vs "emailed" terminology**: the current `sent` status
   means "marked as finalized, no longer draft". Keep that semantic. Add a
   separate concept of "emailed at" (`lastEmailedAt` timestamp) tracking
   the most recent successful email delivery. A user can mark sent without
   emailing, or email without changing status (e.g. re-sending to an
   updated email address).

### Task 2.2 — Payments service

**File:** `packages/server/src/apps/invoices/services/payment.service.ts`
(new)

Functions:
- `listPaymentsForInvoice(invoiceId, tenantId)` → `InvoicePayment[]`
- `recordPayment(input, userId, tenantId)` → `InvoicePayment`
  - Validates `amount > 0`
  - Validates `invoiceId` belongs to tenant
  - Validates `paymentDate <= today`
  - Validates `sum(payments) - sum(refunds) + this amount <= invoice.total`
    for new payments (prevents overpayment); allows refunds up to paid
    amount
  - Inserts the row
  - Recomputes invoice status (see `updateInvoicePaidStatus` below)
  - Optionally sends payment confirmation email to the client if
    `input.sendConfirmation === true` (default false for MVP)
- `updatePayment(paymentId, input, tenantId)` → amount/date/method/reference/notes only; type and invoiceId are immutable
- `deletePayment(paymentId, tenantId)` — hard delete + recomputes status
- `updateInvoicePaidStatus(invoiceId, tenantId)` — internal helper:
  1. Sums all payments (type='payment') minus all refunds (type='refund')
  2. If sum >= invoice.total → status 'paid', stamp `paidAt = NOW()`
  3. Else if sum > 0 → status stays 'sent' (no 'partial' status in v1.10;
     client uses balanceDue to render the partial badge)
  4. Else if sum === 0 → status 'sent' or 'overdue' (whichever applies)
  5. If invoice was already marked paid and refund drops total below
     invoice.total → status reverts to 'sent'

**Replace `markInvoicePaid`** in `invoice.service.ts`: it now records a
single payment for the full amount and calls `updateInvoicePaidStatus`.
Keep the function signature for backward compatibility — callers don't
change.

**Commit:** `feat: implement payment service with balance recomputation`

export const recurringInvoices = pgTable('recurring_invoices', {
  id: uuid('id').primaryKey().defaultRandom(),
  tenantId: uuid('tenant_id').notNull().references(() => tenants.id),
  userId: uuid('user_id').notNull(),
  companyId: uuid('company_id').notNull().references(() => crmCompanies.id),

  description: text('description').notNull(),

### Task 4.2 — Recurring invoices service

**File:** `packages/server/src/apps/invoices/services/recurring-invoice.service.ts`
(new)

Functions:
- `listRecurringInvoices(tenantId, userId, isAdmin)` — filter by user if not admin
- `getRecurringInvoice(id, tenantId)`
- `createRecurringInvoice(input, userId, tenantId)`:
  - Compute `nextRunAt` from `startDate` and `frequency` (first run is
    `startDate` itself, subsequent adds 1 week/month/quarter/year)
- `updateRecurringInvoice(id, input, tenantId)`:
  - If frequency or startDate changed, recompute `nextRunAt`
- `pauseRecurringInvoice(id, tenantId)` — sets `isActive = false`
- `resumeRecurringInvoice(id, tenantId)` — sets `isActive = true`,
  recomputes `nextRunAt` if it's in the past
- `deleteRecurringInvoice(id, tenantId)` — hard delete
- `generateInvoiceFromRecurring(recurringInvoiceId, tenantId)` — the core
  function called by the scheduler:
  1. Load recurring invoice + line items
  2. Create a new `invoices` row with status='draft' (or 'sent' if autoSend)
  3. Copy line items
  4. Compute subtotal/taxAmount/discountAmount/total
  5. Set dueDate = today + paymentTermsDays
  6. Set invoice number via the same auto-numbering used elsewhere
  7. If `autoSend === true`: call `sendInvoiceEmail` from phase 1
  8. Update recurring: `lastRunAt = NOW()`, `runCount += 1`,
     `nextRunAt = computeNext(nextRunAt, frequency)`
  9. If `maxRuns` reached OR `endDate < NOW()`: set `isActive = false`
  10. Return the generated invoice

### Task 5.2 — Reminder eligibility query

**File:** `packages/server/src/apps/invoices/reminder-scheduler.ts` (new)

Exports `startInvoiceReminderScheduler()` and
`stopInvoiceReminderScheduler()`. Hourly cron:

For each tenant with `reminderEnabled = true`:
1. Query invoices that are overdue (dueDate < NOW()) AND have balanceDue > 0
2. For each, determine eligibility:
   - Stage 0 (no reminder yet) and daysSinceDue >= reminder1Days → send
     stage 1
   - Stage 1 and daysSinceDue >= reminder2Days → send stage 2
   - Stage 2 and daysSinceDue >= reminder3Days → send stage 3
   - Stage 3 and daysSinceLastReminder >= endlessReminderDays → send stage 4+
     (endless reminder, doesn't increment stage past 4)
3. For each eligible invoice, call
   `sendInvoiceEmail(..., { template: 'reminder', stage })`
4. Update `lastReminderStage` and `lastReminderAt` on success

**Important**: reminders stop automatically when `balanceDue <= 0` (because
that's a filter condition on the eligibility query). Paying an invoice
cancels the dunning.

Wire into `packages/server/src/index.ts` alongside the recurring
scheduler. Model on `packages/server/src/apps/sign/reminder.ts`.
